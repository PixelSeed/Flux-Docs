    Maya Toolbox 架构分析与升级建议 (资深架构师评估)¶

    本报告基于对 floating_toolbox 现有代码库的深入审计，针对多版本兼容、高扩展性及三方工具集成提出核心改进方案。

    🏗️ 当前架构评估¶

    1. 现有亮点¶

    • 多版本分发: 使用 .mod 条件路径支持不同 Maya 版本的二进制插件加载。
    • 环境隔离: core/maya/ 明确区分了纯 Python 逻辑与 Maya 依赖。
    • 配置安全: ConfigManager 实现了原子写入，有效防止 Maya 崩溃导致的数据损坏。
    • 多模式加载: 支持内置工具、高级包工具以及用户自定义脚本。

    2. 核心痛点¶

    • 加载器硬编码: ToolLoader 内部硬编码了三种加载逻辑，每增加一种工具来源（如云端工具库）都需修改核心代码。
    • 依赖冲突风险: builtin_tools.py 在顶层导入 maya.cmds，导致非 Maya 环境（如单元测试或 CI）直接崩溃。
    • 适配机制缺失: 第三方复杂工具（如 AdvancedSkeleton）缺乏标准入口，集成不够规范。
    • 版本控制粗糙: 插件级别有版本分流，但具体到单个工具缺乏版本限制声明（Manifest）。

    🎯 推荐架构：Provider 插件化 + Manifest 声明化¶

    核心思路是将“工具来源”与其“加载策略”解耦。

    1. 新的目录结构方案¶

    core/
├── providers/              # NEW: 加载器插件系统
│   ├── base_provider.py    # 抽象接口（定义 can_handle, load）
│   ├── builtin_provider.py # 处理内置单文件工具
│   └── package_provider.py # 处理 advanced/ 下的文件夹包
├── tool_manifest.py        # NEW: 工具清单校验逻辑
└── tool_loader.py          # 改造为 Provider 注册表

    2. 声明式清单 (Manifest.json)¶

    为每个复杂工具增加一个 manifest.json： 

    {
  "id": "batch_export",
  "name": "批量导出",
  "version": "1.2.0",
  "requires": {
    "maya": ">=2022",
    "python": ">=3.7",
    "plugins": ["ngSkinTools2"]
  },
  "entry": "tool.py"
}

    📋 改造路线图¶

    阶段 1: 基础加固 (短期)¶

    • 拆分内置工具: 将巨大的 builtin_tools.py 按功能拆分为 modeling.py, rigging.py 等。
    • 延迟导入: 移除所有工具顶层的 import maya.cmds，改为在执行方法内导入。
    • 适配器接入: 为 AdvancedSkeleton 等三方工具补全 tool.py 适配层。

    阶段 2: 架构升级 (中期)¶

    • 实现 Provider 模式: 重构 ToolLoader，通过注册 Provider 的方式扫描路径，满足开闭原则。
    • 引入兼容性检查: 根据 manifest.json 在加载阶段自动隐藏/警告不符合环境要求的工具。

    阶段 3: 统计与优化 (长期)¶

    • 数据解耦: 将 ToolModel 分层为元数据与统计数据（使用频率、最后使用时间），优化序列化效率。
    • 外部 Provider: 支持从 ZIP 包或 URL 获取工具，实现真正的热更新。

    [!TIP] 结论建议：应优先执行阶段 1 的内置工具拆分与延迟导入。这不仅能解决目前测试环境崩溃的问题，也为后续的 Provider 模式打下良好的模块化基础。
