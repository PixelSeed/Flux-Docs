    开发接入¶

    本文档面向要为 Flux 增加新工具、外部工具目录、能力包或自动化测试入口的开发者。

    1. 当前支持的工具来源¶

    工具盒当前支持三类主要来源：

    1. 内置工具
    2. 位于 floating_toolbox/tools/builtin
    3. 包式工具
    4. 位于 floating_toolbox/tools/advanced/<tool_id>
    5. 用户工具
    6. 由工具盒内创建并保存在用户目录中

    额外工具目录会复用包式工具的扫描与加载逻辑。

    2. Quick Tool 与 Managed Capability¶

    Flux 当前正式区分两类工具对象。

    Quick Tool¶

    Quick Tool 面向个人快速收纳与手动执行。

    适合场景：

    • 临时脚本
    • 个人工具
    • 不需要 AI 编排的轻量 Maya 操作
    • 只需要分类、搜索、收藏、钉选和手动运行

    Quick Tool 不需要 tool_contract.json，也不会通过 ExecutionBroker 执行。

    Managed Capability¶

    Managed Capability 面向长期维护、AI 编排、治理、复用和审计。

    适合场景：

    • 希望进入 Scheduler / Flow / AI planning 的工具
    • 希望支持 dry-run、确认门禁、side-effect policy 的工具
    • 希望被导入、审核、发布、废弃、退役和审计的工具
    • 希望成为团队或外部 AI 协作者可理解资产的工具

    Managed Capability 必须以 capability package 方式接入，并提供 tool_contract.json。

    Quick Tool / Managed Capability 的产品模型、UI 表现层区别和后续生命周期产品化路线记录在：

    internal_docs/reference/toolbox_capability_package_spec_v1_zh.md

    3. 推荐的正式接入方式¶

    如果你希望工具长期维护、可迁移、可治理，推荐使用能力包。

    最小目录结构：

    my_tool/
  tool_manifest.json
  tool_contract.json
  tool.py

    更完整的能力包：

    my_tool/
  tool_manifest.json
  tool_contract.json
  governance.json
  cards/
  tool.py
  scripts/
  resources/

    当前最适合作为新工具起点的模板：

    floating_toolbox/tools/templates/capability_package_minimal
floating_toolbox/tools/templates/capability_package_ui_tool

    新 UI 工具默认优先从 capability_package_ui_tool 开始。它表达的是当前 L3 作者路径：FluxDialog 窗口、FTStatusStrip 反馈区、FTActionRow 动作区、tool.py 薄入口、scripts/*_ops.py 业务操作层、scripts/ui_patterns.py 局部 UI pattern、cards 引用 contract action，以及 release review 入口。

    模板当前包含的局部 UI pattern 是：form section、preview result panel、inline error field、action result summary。它们先留在模板内，适合新工具复制后按需删减；只有当多个真实生产工具稳定复用同一结构时，再考虑提升到全局 floating_toolbox.ui.kit。

    模板当前的 UI authoring pattern 与 L3 验收标准记录在：

    internal_docs/reference/toolbox_ui_kit_for_tool_authors_zh.md
internal_docs/reference/core_tool_l2_l3_acceptance_authoring_pattern_zh.md

    也可以直接用 Stage-2 脚手架生成新 UI capability package：

    python scripts/scaffold_capability_package.py --template ui_tool --id my_tool --name "My Tool" --output temp/my_tool

    脚手架会复制 capability_package_ui_tool，并替换 tool_id、显示名、class name、card id、manifest / contract 字段和 scripts/*_ui.py / scripts/*_ops.py 文件名。

    如果输出目录不在 Flux 仓库内，运行 review 前需要让 floating_toolbox 可被 Python 找到，例如：

    $env:PYTHONPATH = "D:\Pycharm_Pro\MayaTool"
python temp/my_tool/scripts/run_capability_release.py review --path temp/my_tool

    本机 Maya smoke 包建议按需重新生成，不作为仓库资产提交。示例：

    $root = "C:\Users\<你自己的用户名>\Documents\maya\prefs\floating_toolbox\test_packages"
python scripts/scaffold_capability_package.py --template ui_tool --id maya_l3_ui_smoke_tool --name "Maya L3 UI Smoke Tool" --output "$root\maya_l3_ui_smoke_tool" --force

    然后在 Flux 的外部工具路径中添加父目录：

    C:\Users\<你自己的用户名>\Documents\maya\prefs\floating_toolbox\test_packages

    该路径只适合作为当前机器的手动验收目录。换电脑时重新生成测试包，或把输出目录改成自己的同步工具库目录。

    4. 当前正式生产工具样板¶

    当前正式内置生产工具集合固定为：

    • custom_color
    • controller_tools
    • smart_joint

    它们的参考方式不同：

    • custom_color：生产完整度样板，适合参考完整 package、contract、cards、governance、dry-run 和资源组织。
    • smart_joint：轻量 FluxDialog 和标准反馈区样板，适合参考新 UI 工具结构。
    • controller_tools：大型 legacy UI 渐进迁移样板，不建议作为新工具 UI 模板直接复制。

    不要从旧 AI demo、历史已删除的 Pro demo、legacy examples 复制新工具结构。新 UI 工具也不要直接复制 controller_tools 的大型 legacy UI；优先复制 capability_package_ui_tool，再按 custom_color 或 smart_joint 补充真实生产细节。

    5. tool_manifest.json¶

    tool_manifest.json 负责工具身份和加载入口。

    它通常声明：

    • id
    • name
    • version
    • author
    • category
    • entry_point
    • Maya 版本兼容性
    • Toolbox 版本兼容性
    • upgrade / compatibility 元数据

    示例：

    {
  "schema_version": "tool_manifest.v1",
  "id": "my_tool",
  "name": "My Tool",
  "version": "1.0.0",
  "author": "someone",
  "category": "Rigging",
  "entry_point": "tool.py",
  "maya_versions": {
    "min": 2022,
    "max": 2026
  },
  "toolbox_versions": {
    "min": "1.0.0"
  }
}

    tool_manifest.json 不定义执行协议。执行协议属于 tool_contract.json。

    6. tool_contract.json¶

    tool_contract.json 是 Managed Capability 的执行协议真相源。

    它声明：

    • tool_id
    • schema version
    • action 列表
    • 每个 action 的参数 schema
    • risk / side effect 语义
    • confirmation 策略
    • dry-run 支持
    • main-thread 要求
    • selection policy
    • returns 结构

    AI、Scheduler、Flow 和 ExecutionBroker 都应该围绕 package contract 理解工具能力。

    7. 入口文件 tool.py¶

    包式工具最终需要有一个可被加载器导入的入口文件。ToolLoader 会根据 manifest 的 entry_point 定位它。

    建议：

    • 顶层尽量不要 import Maya / Qt。
    • Maya 操作放进 lazy execution branch 或 runtime helper。
    • UI 打开使用明确 action，例如 open_ui。
    • 真实执行 action 与 UI action 分开。
    • dry-run 分支不得触发真实 Maya 修改。
    • 返回结构化结果，失败时给出稳定 error_code。

    8. 执行入口为什么要接 Broker¶

    Managed Capability 的正式执行入口是：

    floating_toolbox/execution/runtime/execution_broker.py

    如果你的工具要接入 AI 调度、workflow cards、dry-run、确认门禁、side-effect policy 或审计，它必须对齐：

    • tool_contract.json
    • ExecutionRegistry
    • ExecutionBroker
    • ExecutionGovernancePolicy

    不要把关键写操作只绑定在 UI 点击回调上。

    9. 能力包生命周期¶

    当前能力资产生命周期后端已经具备最小闭环：

    external package
  -> import
  -> validation
  -> pending_review
  -> approved / rejected
  -> active
  -> deprecated
  -> retired

    相关入口：

    • floating_toolbox.import_capability_package(...)
    • floating_toolbox.list_external_import_review_items(...)
    • floating_toolbox.apply_external_import_review_decision(...)
    • floating_toolbox.query_ai_execution_logs(...)

    后续 UI/CLI 产品入口仍会继续打磨，但工具包协议和应用层 UseCase 已经是当前主链路。

    10. 外部工具目录怎么接¶

    如果你不想把工具直接放进仓库，可以：

    1. 将包式工具目录放到外部路径
    2. 在设置页的“工具路径”里添加该目录
    3. 或通过 FLUX_TOOL_PATHS 环境变量提供路径

    这样工具盒会按外部包目录扫描。

    11. 测试入口¶

    推荐使用统一脚本：

    powershell -ExecutionPolicy Bypass -File scripts/test.ps1

    默认会运行一组轻量守卫测试，覆盖架构文档同步、UI kit API、执行 Broker、ToolLoader hard-cut、能力状态、能力资产状态机等主链路。

    运行全部测试：

    powershell -ExecutionPolicy Bypass -File scripts/test.ps1 -All

    只运行架构/文档守卫：

    powershell -ExecutionPolicy Bypass -File scripts/test.ps1 -Architecture

    运行指定测试：

    powershell -ExecutionPolicy Bypass -File scripts/test.ps1 -Targets floating_toolbox/tests/execution_broker_spec.py

    如果当前 shell 的 python 指向 Windows Store placeholder，请指定真实解释器：

    $env:FLUX_TEST_PYTHON = "C:\Path\To\python.exe"
powershell -ExecutionPolicy Bypass -File scripts/test.ps1

    需要 Maya runtime 时，使用 mayapy.exe：

    $env:FLUX_TEST_PYTHON = "C:\Program Files\Autodesk\Maya2026\bin\mayapy.exe"
powershell -ExecutionPolicy Bypass -File scripts/test.ps1

    脚本也支持直接传入解释器：

    powershell -ExecutionPolicy Bypass -File scripts/test.ps1 -Python "C:\Path\To\python.exe"

    12. 什么时候看 internal_docs¶

    如果你只想加一个简单 Quick Tool，看本页通常就够了。

    如果你要做下面这些事情，需要继续看 internal_docs：

    • 接入完整 capability package
    • 对齐 contract / governance / cards 协议
    • 参与核心架构治理
    • 修改执行、AI、workflow 或 asset lifecycle 主链路
