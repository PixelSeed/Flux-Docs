    架构概览（当前实现）¶

    last_verified_commit: e4a50e3 last_verified_date: 2026-07-20 status: active

    本文档面向外部阅读者和新接入开发者，说明当前 Flux 代码如何组织，以及普通工具、标准能力、UI、AI、工作流和执行治理如何连接。更严格的内部架构约束以 internal_docs/active/architecture_constitution_v1_zh.md 为准。

    副作用 authority¶

    系统按资源类型治理副作用：Maya 场景与 Managed Capability 执行由 ExecutionBroker 负责；未来 W3 的 FileEditBroker 负责 CodeChangeCase 计划内的项目工作区文件修改；Case、配置、缓存、artifact 和 Projection 等应用数据持久化仍经 UseCase + Repository Port。后两者不是当前 ExecutionBroker 的职责扩张。

    Maya 操作分为 Passive Observation、Reversible Transactional Probe 和 Persistent Mutation。Probe 即使最终 undo，也不是纯只读；正式 production proof probe 必须进入 ExecutionBroker 的低风险 policy，并记录结构化 probe spec、Scene Revision、恢复复测、content-addressed artifact、Case/plan/tool provenance 和 VerificationRecord。W7 的 production Behavior proof 还必须声明可验证的 expected plug response contract；Deformation proof 只以 object-space vertex delta 声明局部响应，并把整体 mesh transform 单独标为 rigid response。旧 probe_rig_stimulus_response / B0 shadow adapter 仍不得绕过该链直接生成 production effect claim。

    1. 产品定位¶

    Flux 首先是一个 Maya 工具盒，其次是一个面向 AI 时代的工具能力资产管理与复用平台。

    这意味着：

    • 普通用户可以像使用传统工具盒一样添加、分类、搜索、收藏、钉选和运行工具。
    • 标准化工具可以被纳管为能力包，进入 contract 校验、AI 编排、Scheduler、Flow、治理、审计和生命周期管理。
    • AI 负责理解意图、提出候选工具或流程、建议参数；系统负责校验、裁定、执行和记录。

    一句话概括：

    AI 负责提案，系统负责裁定。

    2. 两类工具对象¶

    当前架构正式区分 Quick Tool 和 Managed Capability。它们不是“残缺工具”和“完整工具”的关系，而是面向不同使用场景的两条产品路径。

    2.1 Quick Tool¶

    Quick Tool 面向个人快速收纳和手动执行。

    它可以：

    • 保存脚本或本地工具入口
    • 进入分类、搜索、收藏、钉选、最近使用
    • 通过 UI、Shelf、快捷键或工具盒面板手动运行

    它不会：

    • 要求 tool_contract.json
    • 进入 AI planning
    • 进入 Scheduler / Flow 编排
    • 通过 ExecutionBroker 治理执行
    • 作为标准能力资产进入生命周期审计

    Quick Tool 的典型执行链路：

    UI / Shelf / Quick Panel
  -> ToolboxViewModel
  -> ToolExecutionUseCase
  -> quick_tool_direct
  -> RuntimeTool.execute

    2.2 Managed Capability¶

    Managed Capability 面向 AI 编排、复用、治理、审计和长期维护。

    最小标准能力包需要：

    my_tool/
  tool_manifest.json
  tool_contract.json
  tool.py

    更完整的 L2/L3 能力包通常包含：

    my_tool/
  governance.json
  cards/
  resources/
  scripts/

    Managed Capability 的典型执行链路：

    AI / Scheduler / Flow / Managed UI execution
  -> ExecutionBroker
  -> package contract validation
  -> governance / confirmation / audit
  -> RuntimeTool.execute

    关键规则：

    • tool_manifest.json 负责身份、版本、兼容性和加载入口。
    • tool_contract.json 是执行协议真相源。
    • governance.json 是治理补充策略，不替代 contract。
    • cards/*.json 是可复用样例或工作流入口，必须引用 contract action。
    • 无 package contract 的工具不能通过 ExecutionBroker 执行。

    3. 主干调用链¶

    普通打开工具盒：

    floating_toolbox.show_toolbox()
  -> floating_toolbox.main.show_toolbox()
    -> ToolboxViewModel
    -> ToolboxController
      -> IdlePanel / QuickPanel / FullPanel

    工具执行：

    Panel / Tool List
  -> ToolboxViewModel.execute_tool()
  -> ToolExecutionUseCase
     -> managed capability: ExecutionBroker
     -> quick tool: quick_tool_direct
  -> execution log
  -> refresh recent tools projection

    AI / Scheduler / Flow 执行：

    AI planner / Scheduler V2 / Flow runner
  -> ExecutionRegistry
  -> package tool_contract.json
  -> ExecutionBroker
  -> governance policy / validation / confirmation
  -> RuntimeTool.execute

    4. 用户可感知的核心层¶

    UI 层¶

    • 包入口在 floating_toolbox.__init__
    • Maya 级入口在 floating_toolbox.main
    • 面板切换由 ToolboxController 负责
    • UI 业务状态主要集中在 ToolboxViewModel
    • FullPanel 是当前完整工具盒主入口
    • FluxDialog、FTActionRow、FTStatusStrip 等组件位于 floating_toolbox/ui/kit

    工具发现与组织层¶

    • 内置工具目录：floating_toolbox/tools/builtin
    • 正式高级能力包目录：floating_toolbox/tools/advanced
    • 作者模板目录：floating_toolbox/tools/templates
    • ToolLoader 负责扫描、校验和加载工具
    • 外部工具目录复用 package 扫描策略
    • ToolScanResult 返回工具列表、capability reports 和迁移信息

    执行与治理层¶

    • execution/runtime/execution_broker.py 是当前治理执行入口
    • FileEditBroker 是 W3 目标 authority，用于 CodeChangeCase 计划内的项目工作区文件修改；当前尚未实现，现有 ProjectEditService 不能被描述为该 authority
    • execution/contracts/registry.py 负责发现 package contract
    • execution/contracts/protocol_resolution.py 负责协议来源归一
    • infrastructure/projections/ 保存 UI、菜单、查询等可重建读模型
    • Broker 会处理工具查找、contract 校验、参数校验、确认门禁、side-effect policy、dry-run / impact preview、执行结果归一和治理 trace

    AI 与编排层¶

    • AI provider 选择走 ProviderPolicy / ProviderRouter
    • AI catalog 只应暴露标准能力
    • Managed Capability 与 Maya 场景操作最终回到 ExecutionBroker；CodeChangeCase 的计划内工作区文件修改将在 W3 迁移至 FileEditBroker
    • Scheduler V2 是当前 AI 任务调度入口
    • Workflow cards 和 template 支持位于 floating_toolbox/workflow

    能力资产生命周期层¶

    能力包不是 loose script，而是可治理资产。当前生命周期基础包括：

    • 外部能力包导入
    • package validation
    • review queue item
    • approve / reject
    • active / deprecated / retired 状态转换
    • audit event
    • version snapshot
    • major version bump warning
    • package discovery cache invalidation

    核心代码包括：

    • floating_toolbox/application/capability/capability_package_import.py
    • floating_toolbox/application/capability/external_import_review_query_service.py
    • floating_toolbox/application/capability/external_import_review_action_usecase.py
    • floating_toolbox/application/capability/capability_asset_lifecycle_usecase.py
    • floating_toolbox/infrastructure/repositories/capability_asset_config_repository.py

    配置与持久化层¶

    • core/config_manager.py 负责主配置读写与迁移
    • core/storage_paths.py 负责版本级与共享级路径
    • 新业务写路径优先通过 application + domain + infrastructure，不继续把规则堆进 UI 或 legacy manager

    5. 当前正式生产工具¶

    当前正式内置生产工具集合固定为：

    • floating_toolbox/tools/advanced/custom_color
    • floating_toolbox/tools/advanced/controller_tools
    • floating_toolbox/tools/advanced/smart_joint

    三者都应保持：

    • tool_manifest.json
    • tool_contract.json
    • governance.json
    • cards/
    • tool.py
    • lazy Maya / Qt runtime imports
    • UI 作者路径逐步靠拢 FluxDialog 和 UI kit

    它们的当前角色不同：

    • custom_color 是生产完整度样板。
    • smart_joint 是轻量 FluxDialog / 标准反馈区样板。
    • controller_tools 是大型 legacy UI 渐进迁移样板，不建议作为新工具 UI 模板直接复制。

    历史上的 export_pro、rename_pro、select_pro 和旧 AI demo / examples 已不属于当前正式生产工具集合；如在旧文档或本地缓存里看到这些名字，应按历史已删除路径处理。

    6. 当前目录语义¶

    floating_toolbox/
  ui/                 # Presentation, panels, dialogs, UI kit
  application/        # UseCase / query service / orchestration ports
  domain/             # Business rules, states, entities, value objects
  infrastructure/     # Repository adapters and external integration
  execution/          # Contract registry, governance policy, validation
    runtime/          # ExecutionBroker and runtime request/result objects
  ai/                 # Planning, provider routing, AI-assisted generation
  workflow/           # Cards, templates, Scheduler helpers
  infrastructure/
    projections/      # Rebuildable read models for UI/menu/query surfaces
  core/               # Legacy managers and shared runtime utilities
  tools/
    builtin/          # Lightweight built-in tools
    advanced/         # Current formal production capability packages
    templates/        # Capability package authoring templates

    core/ 当前仍是历史实现聚集区。新写路径优先进入 application/domain/infrastructure，再由 adapter 适配旧 manager 或存储实现。

    7. 你最可能需要关心的代码入口¶

    1. 打开工具盒：floating_toolbox.show_toolbox
    2. 打开完整面板：floating_toolbox.show_full_panel
    3. 热键打开完整面板：floating_toolbox.show_full_panel_from_hotkey
    4. 打开 AI 任务调度：floating_toolbox.show_ai_task_dialog
    5. Scheduler V2 入口：floating_toolbox.show_ai_task_scheduler_v2_dialog
    6. 导入能力包：floating_toolbox.import_capability_package
    7. 审核外部导入：floating_toolbox.list_external_import_review_items
    8. 应用审核决定：floating_toolbox.apply_external_import_review_decision
    9. 查询执行日志：floating_toolbox.query_ai_execution_logs
    10. 入口实现：floating_toolbox/main.py
    11. UI 执行 UseCase：floating_toolbox/application/tool/tool_execution_usecase.py
    12. 执行治理入口：floating_toolbox/execution/runtime/execution_broker.py
    13. 执行协议注册：floating_toolbox/execution/contracts/registry.py
    14. 执行协议归一：floating_toolbox/execution/contracts/protocol_resolution.py
    15. 工具加载：floating_toolbox/infrastructure/tool_loading/tool_loader.py
    16. 配置读写：floating_toolbox/core/config_manager.py

    9. 测试入口¶

    推荐从仓库根目录运行：

    powershell -ExecutionPolicy Bypass -File scripts/test.ps1

    如果系统 python 指向 Windows Store placeholder，请显式指定真实解释器：

    $env:FLUX_TEST_PYTHON = "C:\Path\To\python.exe"
powershell -ExecutionPolicy Bypass -File scripts/test.ps1

    如果测试依赖 Maya runtime，请使用 mayapy.exe：

    $env:FLUX_TEST_PYTHON = "C:\Program Files\Autodesk\Maya2026\bin\mayapy.exe"
powershell -ExecutionPolicy Bypass -File scripts/test.ps1
